# Analyze Feasibility Skill

## User Input

```text
$ARGUMENTS
```

You **MUST** consider the user input before proceeding (if not empty).

## Outline

### 1. 需求理解与解析

从用户输入中提取关键信息：
- **核心功能**: 用户想要实现什么？
- **业务价值**: 为什么需要这个功能？
- **用户场景**: 谁会使用？在什么情况下使用？
- **约束条件**: 有哪些明确的限制或要求？

如果用户输入过于简短或模糊，使用 `AskUserQuestion` 工具询问关键细节（最多 3 个问题）。

### 2. 技术可行性分析

#### 2.1 代码库兼容性检查

1. **读取项目结构**:
   ```bash
   # 检查主要技术栈
   cat Cargo.toml package.json 2>/dev/null | head -20

   # 检查现有架构
   ls -la src/ 2>/dev/null
   ```

2. **识别相关模块**:
   - 使用 `Grep` 工具搜索相关代码模式
   - 检查是否存在类似功能的实现
   - 识别需要修改的核心文件

3. **依赖分析**:
   - 检查是否需要新的外部依赖
   - 评估依赖的成熟度和维护状态
   - 识别潜在的依赖冲突

#### 2.2 架构影响评估

- **模块耦合度**: 需要修改多少个模块？
- **API 变更**: 是否需要修改公共接口？
- **数据迁移**: 是否需要数据库 schema 变更？
- **向后兼容性**: 是否会破坏现有功能？

### 3. 实现复杂度评估

使用以下维度评分（1-5 分，5 分最复杂）：

| 维度 | 评分 | 说明 |
|------|------|------|
| **代码量** | ? | 预估需要编写/修改的代码行数 |
| **技术难度** | ? | 算法复杂度、新技术学习成本 |
| **测试复杂度** | ? | 单元测试、集成测试的覆盖难度 |
| **风险等级** | ? | 对现有系统的潜在影响 |
| **依赖复杂度** | ? | 外部依赖的数量和复杂度 |

**总体复杂度** = (各维度评分之和) / 5

- **1.0-2.0**: 简单（1-2 天）
- **2.1-3.5**: 中等（3-5 天）
- **3.6-5.0**: 复杂（1-2 周或更长）

### 4. 风险识别

列出潜在风险点：
- **技术风险**: 不确定的技术方案、性能瓶颈
- **业务风险**: 需求不明确、范围蔓延
- **集成风险**: 与现有系统的兼容性问题
- **安全风险**: 潜在的安全漏洞（XSS、SQL 注入等）

### 5. 实现方案建议

提供 **2-3 个可选方案**，每个方案包括：

#### 方案 A: [方案名称]
- **核心思路**: 简要描述实现思路
- **优点**: 列出主要优势
- **缺点**: 列出主要劣势
- **预估工时**: X 天
- **推荐指数**: ⭐⭐⭐⭐⭐ (5 星最高)

#### 方案 B: [方案名称]
...

### 7. 输出总结

向用户展示：
1. **可行性结论**: 一句话总结（可行/有风险但可行/不建议）
2. **复杂度评级**: 简单/中等/复杂
3. **推荐方案**: 简要说明推荐的实现方案
4. **下一步行动**:
   - 如果可行：建议使用 `/create-issue` 创建 GitHub Issue
   - 如果有风险：建议先解决哪些问题
   - 如果不可行：说明原因和替代方案

## 注意事项

1. **保持客观**: 基于代码库实际情况分析，不要过度乐观或悲观
2. **量化评估**: 尽可能使用具体数据（代码行数、文件数量、依赖数量）
3. **考虑维护性**: 不仅评估实现难度，还要考虑长期维护成本
4. **遵循原则**: 分析时应用 SOLID、KISS、YAGNI、DRY 原则
5. **安全优先**: 特别关注潜在的安全风险

## 示例用法

```bash
# 用户输入
/analyze-feasibility 我想添加一个用户登录功能，支持邮箱和密码登录

# AI 执行流程
1. 解析需求 → 识别核心功能：用户认证系统
2. 检查代码库 → 搜索现有认证相关代码
3. 评估复杂度 → 需要实现：注册、登录、会话管理、密码加密
4. 识别风险 → 安全风险（密码存储）、会话管理复杂度
5. 提供方案 → 方案 A: 使用 JWT + bcrypt；方案 B: 使用 OAuth2
6. 生成报告 → 写入 feasibility-report.md
7. 输出总结 → "可行，中等复杂度，推荐方案 A，预估 3-5 天"
```
