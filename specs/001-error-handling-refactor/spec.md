# Feature Specification: 统一错误处理与展示重构

**Feature Branch**: `001-error-handling-refactor`
**Created**: 2026-03-09
**Status**: Draft
**Input**: User description: "我想要重构 rust 端和前端的错误处理与展示,确保所有错误都有自己的 code,而不是通过判断错误的 message 来显示错误信息,自定义 AppError 来区分不同的错误。前端要使用shadcn ui的Sonner来显示错误提示而不是将错误信息嵌入到 ui 组件中。前端要有统一的错误处理,而不是每个模块各自处理。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 后端错误代码标准化 (Priority: P1)

当系统发生错误时,后端返回结构化的错误响应,包含唯一的错误代码和人类可读的消息,使前端能够准确识别和处理不同类型的错误。

**Why this priority**: 这是整个错误处理重构的基础。没有标准化的错误代码,前端无法实现统一的错误处理逻辑。这是最核心的改进,直接影响系统的可维护性和用户体验。

**Independent Test**: 可以通过触发各种后端错误场景(如验证失败、资源不存在、权限不足等),验证每个错误都返回唯一的错误代码和一致的响应结构。

**Acceptance Scenarios**:

1. **Given** 用户提交了无效的表单数据, **When** 后端验证失败, **Then** 返回包含错误代码(如 `VALIDATION_ERROR`)和具体字段错误信息的结构化响应
2. **Given** 用户尝试访问不存在的资源, **When** 后端处理请求, **Then** 返回包含错误代码(如 `RESOURCE_NOT_FOUND`)和资源标识的结构化响应
3. **Given** 用户没有权限执行某操作, **When** 后端检查权限, **Then** 返回包含错误代码(如 `PERMISSION_DENIED`)的结构化响应
4. **Given** 系统发生内部错误, **When** 后端捕获异常, **Then** 返回包含错误代码(如 `INTERNAL_ERROR`)的结构化响应,不暴露敏感信息

---

### User Story 2 - 前端统一错误处理机制 (Priority: P2)

前端建立统一的错误处理层,自动拦截所有 API 错误响应,根据错误代码执行相应的处理逻辑,无需在每个组件中重复编写错误处理代码。

**Why this priority**: 在后端错误标准化之后,前端需要统一的机制来消费这些错误。这消除了代码重复,确保错误处理的一致性,是实现良好用户体验的关键。

**Independent Test**: 可以通过模拟各种 API 错误响应,验证统一错误处理层能够正确拦截、解析错误代码,并触发相应的处理逻辑(如显示通知、跳转登录页等)。

**Acceptance Scenarios**:

1. **Given** 任何 API 调用返回错误响应, **When** 前端接收到响应, **Then** 统一错误处理层自动拦截并解析错误代码
2. **Given** 错误代码为认证相关(如 `AUTH_TOKEN_EXPIRED`), **When** 统一错误处理层处理, **Then** 自动清除本地认证信息并跳转到登录页
3. **Given** 错误代码为业务逻辑错误, **When** 统一错误处理层处理, **Then** 根据错误代码显示相应的用户友好提示
4. **Given** 错误代码为网络错误, **When** 统一错误处理层处理, **Then** 显示网络连接问题提示并提供重试选项

---

### User Story 3 - Toast 通知展示错误 (Priority: P3)

用户在操作过程中遇到错误时,系统通过 Toast 通知(使用 Sonner 组件)以非侵入式的方式展示错误信息,而不是在 UI 组件内嵌入错误消息。

**Why this priority**: 这是用户体验层面的改进。在前两个优先级完成后,这一层提供了一致、优雅的错误展示方式,提升了整体用户体验。

**Independent Test**: 可以通过触发各种错误场景,验证 Toast 通知能够正确显示错误信息,包括通知的样式、持续时间、可关闭性等。

**Acceptance Scenarios**:

1. **Given** 用户执行操作导致错误, **When** 统一错误处理层处理错误, **Then** 在屏幕右上角显示 Toast 通知,包含错误标题和描述
2. **Given** Toast 通知正在显示, **When** 用户点击关闭按钮或等待自动消失, **Then** Toast 通知平滑消失
3. **Given** 多个错误同时发生, **When** 统一错误处理层处理, **Then** Toast 通知按顺序或堆叠显示,不会相互覆盖
4. **Given** 错误严重程度不同, **When** Toast 通知显示, **Then** 根据严重程度使用不同的视觉样式(如颜色、图标)

---

### Edge Cases

- 当后端返回的错误响应格式不符合预期时,前端如何处理?
- 当网络完全断开,无法获取任何响应时,如何向用户展示错误?
- 当同一时间发生大量错误(如批量操作失败)时,如何避免 Toast 通知泛滥?
- 当错误代码在前端未定义映射时,如何提供降级处理?
- 当用户快速连续触发相同错误时,如何避免重复显示相同的 Toast?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 后端 MUST 为每种错误类型定义唯一的错误代码(如 `VALIDATION_ERROR`, `RESOURCE_NOT_FOUND`, `PERMISSION_DENIED` 等)
- **FR-002**: 后端 MUST 返回结构化的错误响应,包含错误代码、人类可读的消息、时间戳和可选的详细信息字段
- **FR-003**: 后端 MUST 通过自定义 AppError 类型统一处理所有错误,确保错误响应格式一致
- **FR-004**: 前端 MUST 实现统一的错误拦截机制,自动处理所有 API 错误响应
- **FR-005**: 前端 MUST 根据错误代码执行相应的处理逻辑,而不是依赖错误消息文本
- **FR-006**: 前端 MUST 使用 Sonner Toast 组件展示错误通知,不在 UI 组件内嵌入错误消息
- **FR-007**: 前端 MUST 为常见错误代码提供用户友好的本地化消息映射
- **FR-008**: 系统 MUST 对未定义的错误代码提供降级处理,显示通用错误提示
- **FR-009**: 系统 MUST 区分不同严重程度的错误(如信息、警告、错误、致命错误),并在 UI 中使用不同的视觉样式
- **FR-010**: 系统 MUST 防止相同错误在短时间内重复显示 Toast 通知

### Key Entities

- **AppError**: 后端自定义错误类型,包含错误代码、消息、时间戳、严重程度、可选的详细信息和堆栈跟踪(仅开发环境)
- **ErrorResponse**: 后端返回给前端的标准化错误响应结构,包含 code、message、timestamp、details 等字段
- **ErrorHandler**: 前端统一错误处理器,负责拦截 API 错误、解析错误代码、执行处理逻辑
- **ErrorCodeMap**: 前端错误代码到用户友好消息的映射表,支持国际化

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 所有后端错误响应都包含唯一的错误代码,100% 的错误可通过代码而非消息文本识别
- **SC-002**: 前端错误处理代码集中在统一的错误处理层,各业务模块中的错误处理代码减少 80% 以上
- **SC-003**: 用户遇到错误时,90% 的情况下能看到清晰、友好的错误提示,而不是技术性的错误消息
- **SC-004**: 开发人员添加新的错误类型时,只需在后端定义错误代码和在前端添加映射,无需修改业务逻辑代码
- **SC-005**: 错误通知以非侵入式的 Toast 形式展示,不阻塞用户继续操作,用户满意度提升
