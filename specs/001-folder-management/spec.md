# Feature Specification: Vault 文件夹管理功能

**Feature Branch**: `001-folder-management`
**Created**: 2026-03-11
**Status**: Draft
**Input**: User description: "feat: 实现 vault 页面文件夹管理功能（新增、修改、删除）"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 创建新文件夹组织密码 (Priority: P1)

用户需要创建新文件夹来分类管理不同类型的密码（如工作账号、个人账号、银行账号等），使密码库更有条理。

**Why this priority**: 这是最基础的功能，没有创建功能就无法开始使用文件夹系统。这是 MVP 的核心价值。

**Independent Test**: 用户可以在 vault 页面点击"新建文件夹"按钮，输入文件夹名称后保存，新文件夹立即出现在文件夹列表中，可以用于组织 cipher。

**Acceptance Scenarios**:

1. **Given** 用户在 vault 页面，**When** 点击"新建文件夹"按钮并输入名称"工作账号"，**Then** 系统创建新文件夹并显示在文件夹列表中
2. **Given** 用户输入文件夹名称，**When** 名称为空或仅包含空格，**Then** 系统显示错误提示"文件夹名称不能为空"
3. **Given** 用户输入文件夹名称，**When** 名称与现有文件夹重复，**Then** 系统显示错误提示"文件夹名称已存在"
4. **Given** 用户正在创建文件夹，**When** 点击取消按钮，**Then** 系统关闭创建对话框且不创建文件夹

---

### User Story 2 - 重命名文件夹 (Priority: P2)

用户需要修改文件夹名称以更好地反映其内容或适应组织结构变化。

**Why this priority**: 这是重要的维护功能，但不是初始使用的必需功能。用户可以先创建正确名称的文件夹。

**Independent Test**: 用户可以选择现有文件夹，点击"重命名"选项，输入新名称后保存，文件夹名称立即更新，其中的 cipher 保持不变。

**Acceptance Scenarios**:

1. **Given** 用户选择了一个文件夹，**When** 点击"重命名"并输入新名称"个人账号"，**Then** 文件夹名称更新为"个人账号"
2. **Given** 用户正在重命名文件夹，**When** 输入的新名称为空，**Then** 系统显示错误提示"文件夹名称不能为空"
3. **Given** 用户正在重命名文件夹，**When** 输入的新名称与其他文件夹重复，**Then** 系统显示错误提示"文件夹名称已存在"
4. **Given** 文件夹包含多个 cipher，**When** 重命名文件夹，**Then** 所有 cipher 仍然保留在该文件夹中

---

### User Story 3 - 删除不需要的文件夹 (Priority: P3)

用户需要删除不再使用的文件夹以保持 vault 整洁。

**Why this priority**: 这是清理功能，优先级最低。用户可以暂时保留空文件夹或不使用的文件夹。

**Independent Test**: 用户可以选择文件夹，点击"删除"选项，确认后文件夹从列表中移除。如果文件夹包含 cipher，系统会提示用户处理这些 cipher。

**Acceptance Scenarios**:

1. **Given** 用户选择了一个空文件夹，**When** 点击"删除"并确认，**Then** 文件夹从列表中移除
2. **Given** 用户选择了包含 cipher 的文件夹，**When** 点击"删除"，**Then** 系统显示警告"此文件夹包含 X 个密码项，删除后这些项将移至未分类"
3. **Given** 用户看到删除警告，**When** 确认删除，**Then** 文件夹被删除，其中的 cipher 移至未分类或根目录
4. **Given** 用户看到删除确认对话框，**When** 点击取消，**Then** 文件夹保持不变

---

### Edge Cases

- 当用户尝试创建超过 100 个文件夹时会发生什么？（假设合理上限为 100 个文件夹）
- 当文件夹名称包含特殊字符（如 `/`, `\`, `<`, `>` 等）时系统如何处理？
- 当用户在网络断开时执行文件夹操作会发生什么？
- 当多个设备同时修改同一文件夹时如何处理冲突？
- 当文件夹名称超过 255 个字符时系统如何处理？

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须允许用户在 vault 页面创建新文件夹
- **FR-002**: 系统必须验证文件夹名称不为空且不与现有文件夹重复
- **FR-003**: 系统必须允许用户重命名现有文件夹
- **FR-004**: 系统必须允许用户删除文件夹
- **FR-005**: 系统必须在删除包含 cipher 的文件夹前显示警告并说明影响
- **FR-006**: 系统必须在删除文件夹后将其中的 cipher 移至未分类区域
- **FR-007**: 系统必须在所有文件夹操作后立即更新 UI 显示
- **FR-008**: 系统必须持久化所有文件夹操作（创建、重命名、删除）
- **FR-009**: 系统必须在文件夹操作失败时显示清晰的错误消息
- **FR-010**: 系统必须支持文件夹名称包含中文、英文、数字和常见标点符号
- **FR-011**: 系统必须限制文件夹名称长度在 1-255 个字符之间
- **FR-012**: 系统必须过滤或转义文件夹名称中的危险字符（如路径分隔符）

### Key Entities

- **Folder（文件夹）**: 用于组织 cipher 的容器，包含属性：唯一标识符、名称、创建时间、修改时间、所属用户
- **Cipher（密码项）**: 存储在文件夹中的密码条目，与文件夹是多对一关系（一个 cipher 属于一个文件夹，一个文件夹包含多个 cipher）

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户可以在 3 秒内完成创建新文件夹操作（从点击按钮到文件夹出现在列表）
- **SC-002**: 用户可以在 3 秒内完成重命名文件夹操作
- **SC-003**: 用户可以在 5 秒内完成删除文件夹操作（包括确认步骤）
- **SC-004**: 系统支持至少 100 个文件夹而不影响性能
- **SC-005**: 90% 的用户能在首次尝试时成功完成文件夹操作
- **SC-006**: 文件夹操作的错误率低于 5%（因网络或系统错误导致的失败）
- **SC-007**: 用户对文件夹管理功能的满意度达到 4/5 分或以上

## Assumptions

- 用户已登录并有权访问 vault
- 系统已有基础的 cipher 管理功能
- 文件夹是扁平结构，不支持嵌套（如需嵌套需另行说明）
- 删除文件夹时，cipher 默认移至"未分类"区域而非被删除
- 文件夹操作需要网络连接（假设为在线应用，如为离线应用需另行说明）
- 文件夹数量上限为 100 个（基于常见密码管理器的合理限制）
- 文件夹名称支持 Unicode 字符以适应国际化需求

## Dependencies

- 现有的 vault 页面和 cipher 管理功能
- 用户认证和授权系统
- 数据持久化层（数据库或存储服务）
- 网络连接（用于同步操作）

## Out of Scope

- 文件夹嵌套功能（子文件夹）
- 文件夹拖拽排序
- 文件夹颜色或图标自定义
- 文件夹共享功能
- 批量文件夹操作（批量删除、批量重命名）
- 文件夹导入/导出功能
- 文件夹访问权限控制（假设所有文件夹对用户可见）
